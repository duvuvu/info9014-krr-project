\section{Annotating the Information System}

This section describes how the relational database is mapped to RDF triples using R2RML~\cite{r2rml}, producing an instance of the CineExplorer knowledge graph conforming to the ontology described in the previous section.

\subsection{Approach}

R2RML (RDB to RDF Mapping Language)~\cite{r2rml} is used as the mapping language. R2RML is a W3C recommendation that defines how relational tables and SQL queries can be mapped to RDF graphs. The mapping file \texttt{cineexplorer\_mapping.ttl} is serialized in Turtle syntax and processed by the \texttt{r2rml.jar} tool used in the lab sessions. The generated RDF output is written to \texttt{output/cineexplorer\_kg.ttl} in Turtle format.

The mapping is executed with the following command from the project root:

\begin{verbatim}
java -jar tools/r2rml/r2rml.jar mapping/mapping.properties
\end{verbatim}

where \texttt{mapping.properties} provides the JDBC connection string, credentials, input mapping file path, and output file path.

The mapping strategy follows the conceptual structure of the ontology rather than directly mirroring the table structure. In particular, the mapping is organized around the main ontology entities (\texttt{CreativeWork}, \texttt{Person}, \texttt{Participation}, and \texttt{Genre}) and around a small number of additional triples maps used to enrich these entities with structural or descriptive information.

\subsection{IRI Strategy}

A consistent IRI strategy was defined to ensure that each entity has a stable identifier. IRIs are based on the IMDb identifiers already present in the database, rather than on surrogate numeric keys generated by our schema. This ensures stable identifiers and reuse of widely recognized identifiers

The main IRI templates used are:

\begin{itemize}
    \item \textbf{Creative works} (\texttt{Film}, \texttt{Series}, \texttt{Episode}):\\
    \texttt{https://example.org/cineexplorer/title/\{TITLE\_ID\}}\\
    where \texttt{TITLE\_ID} is the IMDb \texttt{tconst} (e.g., \texttt{tt0000001}).

    \item \textbf{Persons}:\\
    \texttt{https://example.org/cineexplorer/person/\{TALENT\_ID\}}\\
    where \texttt{TALENT\_ID} is the IMDb \texttt{nconst} (e.g., \texttt{nm0000001}).

    \item \textbf{Genres}:\\
    \texttt{https://example.org/cineexplorer/genre/\{GENRE\_ID\}}\\
    where \texttt{GENRE\_ID} is the internal genre identifier.

    \item \textbf{Participation nodes}:\\
    \texttt{https://example.org/cineexplorer/participation/\{TITLE\_ID\}/\{TALENT\_ID\}/\{ORD\}}\\
    where \texttt{ORD} is the credit order from the \texttt{title\_principal} table.
\end{itemize}

For genres, an internal identifier is used because the source data does not provide IMDb-style identifiers for genre instances. The participation IRI is composite because a participation record is context-dependent: it is identified by the work, the person, and the credit order.

This strategy avoids artificial blank nodes for the main entities and ensures that resources can be referenced consistently across triples maps.

\subsection{Mapping Structure}

The mapping is organized by entity type and by the role played by each triples map in the generation of the RDF graph.

\paragraph{Creative works.}
Three triples maps generate the three subclasses of \texttt{CreativeWork}. The distinction between \texttt{Film} and \texttt{Series} is based on the \texttt{CONTENT\_TYPE\_ID} column in the \texttt{title} table. \texttt{Film} includes non-episodic works such as movies, short films, TV movies, TV shorts, TV specials, and videos. \texttt{Series} includes TV series and TV mini-series. In both cases, titles that appear in \texttt{title\_episode} are excluded in order to avoid misclassifying episodic content.

The \texttt{Episode} triples map is based directly on the join between \texttt{title} and \texttt{title\_episode}. This is more robust than relying only on content type, because episodic membership is explicitly represented in the relational schema. The mapping generates the relation \texttt{partOfSeries} together with \texttt{seasonNumber} and \texttt{episodeNumber}.

\paragraph{Persons.}
All rows from the \texttt{talent} table are mapped to \texttt{ce:Person} with their descriptive data properties. Role-specific subclass membership (\texttt{Actor}, \texttt{Director}, \texttt{Writer}, \texttt{Composer}, \texttt{Editor}) is handled through separate triples maps based on joins between \texttt{talent\_role} and \texttt{role}. Since these maps share the same IRI template, the resulting triples are merged on the same subject, allowing one person to belong to multiple role-specific subclasses.

\paragraph{Participation.}
The \texttt{Participation} triples map selects from \texttt{title\_principal} joined with \texttt{category} and generates one participation node per credit record. Each node is linked to its person via \texttt{ce:playedBy} and to its work via \texttt{ce:participatesIn}. The category name is stored as \texttt{ce:participationRole}, while the free-text job description is stored as \texttt{ce:participationProperties}.

Character names, which were normalized into the separate \texttt{principal\_role} table during Milestone~1, are handled by a dedicated triples map targeting the same participation IRI template. This adds \texttt{ce:characterName} triples without duplicating participation nodes.

\paragraph{Genre links.}
Genre instances are mapped from the \texttt{genre} table. The relation between titles and genres is generated from \texttt{title\_genre} through \texttt{ce:hasGenre}. This keeps genre resources separate while preserving the many-to-many relation present in the database.

\paragraph{Known-for links.}
The \texttt{talent\_title} table, which records notable works associated with a person, is mapped to \texttt{ce:knownFor} links from person IRIs to title IRIs.

\paragraph{Additional traversal-oriented properties.}
Besides the participation-centered representation, the mapping also materializes several direct properties between persons and creative works, such as \texttt{workedFor} and \texttt{employed}, together with more specific sub-properties such as \texttt{actedIn}, \texttt{directed}, \texttt{wrote}, \texttt{edited}, \texttt{composedFor}, and \texttt{played}. These properties are redundant from a purely normalization-oriented point of view, since they can be recovered through \texttt{Participation}, but they make graph traversal and query writing more convenient.

Similarly, \texttt{workedWith} is included as a traversal-oriented relation between persons who are connected through participation in the same creative work. This relation is useful for graph exploration use cases such as Bacon-style path analysis.

\paragraph{Language and region metadata.}
Language and region information is derived from \texttt{title\_aka} together with the lookup tables \texttt{language} and \texttt{region}. Rather than introducing standalone resources for these entities, the mapping attaches them as descriptive literals to \texttt{CreativeWork} instances through the datatype properties \texttt{ce:language} and \texttt{ce:region}. This is consistent with the ontology, where language and region are modeled as descriptive attributes rather than as independent conceptual classes.
